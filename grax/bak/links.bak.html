# Links

As we saw in the last section, URIs can be kept in the Grax.Schemas property values for the data properties as values with the :iri datatype. But usually you aren't particularly interested on the URI, but on the description of the identified resource, you'll want to use them as object properties and map them to nested structs with the RDF descriptions of the RDF resources (nodes).

As we've said, object properties are still Grax.Schema properties that are mapped to fields of an Elixir struct. But the values of these properties are mapped differently. As opposed to the values of data properties, which are mapped from RDF literal/XSD datatypes to values of simple Elixir datatypes, the value of object properties are Grax.Schema structs, nested recursively to build up a tree structure under the root resource.

Before we get back to our graph diagram example, let's make a change in our diagram notation. We want to differentiate between leaf nodes with simple values and RDF literals from resource nodes whose description can itself be put into a Grax.Schema struct. So, in the following graph diagrams Grax.Schema structs will be represented as diamond nodes. The URI in the node is the value of the __id__ field of the struct, while the other fields are the properties on the outgoing edges of the Grax.Schema struct. So, the whole tree of the root node and its direct children are the Grax.Schema struct. Think of Grax.Schema structs as root nodes holding Elixir mappings of a subset of the RDF graph with statements directly or indirectly associated with our root node.

If we put the :Address1 resource as our first object property into the picture, we would get this if we would map the identifier to an ordinary property in our schema.

The problem here is that we aren't actually interested in the identifier, in this case a blank node identifier, but the description of the resource it identifies. So, we'll embed the Grax.Schema struct directly as value of the respective object property.

The object properties of RDF resources are represented on Grax.Schemas as links, which are the properties of an RDF resource (node) linking to other RDF resources (nodes). They map the description of resources linked with this property in a RDF graph into a Grax.Schema with the description of this resource which gets embedded into the Grax.Schema of the linking resource. So, the links allow us to traverse the nodes of a graph, but as we're accessing the graph through the Grax.Schema of a particular resource, this traversal happens by accessing it as a tree structure down from a root resource and its fields of nested Grax.Schema structs.

A Grax link can be defined in a Grax schema definition with the link/3 macro. The first two arguments are again for the name and IRI of the property. The :type option however has a different meaning and is no longer optional on link properties. It must be the module name of another Grax.Schema struct.

defmodule User do
  use Grax

  alias NS.{SchemaOrg, FOAF}

  schema do
    property name: SchemaOrg.name, type: :string, required: true
    property email: SchemaOrg.email, type: [:string], required: true
    property age: FOAF.age, type: :integer
    property password: nil
    
    link address: SchemaOrg.address, type: Address
  end
end

defmodule Address do
  use Grax

  alias NS.SchemaOrg

  schema SchemaOrg.PostalAddress do
    property country: SchemaOrg.addressCountry, type: :string
    property city: SchemaOrg.addressLocality, type: :string
    property street: SchemaOrg.streetAddress, type: :string
  end
end

But just as for data properties single struct values are assumed unless the module name is put in square brackets. This allows us to map another of the remaining object properties of our example: the foaf:friends property is mapped to a list of instances of the Post schema struct.

defmodule User do
  use Grax

  alias NS.{SchemaOrg, FOAF}

  schema do
    property name: SchemaOrg.name, type: :string, required: true
    property email: SchemaOrg.email, type: [:string], required: true
    property age: FOAF.age, type: :integer
    property password: nil
    
    link address: PostalAddress
    link friends: FOAF.friend, type: [User]
  end
end

So, in this simplistic example where the address resource consist only of a couple data properties ...

But of course the complexities of mappings of relational associations don't just disappear completely. We need somehow deal with the possible complexities of the associations between the entities of the domain of our application. You might have already asked yourself how the recursive traversal of the graph for loading the nested schema of a root node is done. There are still some problems we have to deal with as we can see on our friends link: how many levels of friends do we want to load and how do we handle circles like in the link for the posts property which is bidirectional linked resource schema.

# Preloading

Preloading is the operation of populating a Grax.Schema struct by loading (mapping) the RDF descriptions of linked resources from a RDF graph into a tree structure over the linked property fields of a Grax.Schema struct as recursive nestings.

There are potentially several useful preloading strategies, which might be implemented in possible future versions. For now, the only preloading strategy supported is the depth preloading strategy, where all of the properties and links up to a specified recursive depth.

The default behaviour for how deep the links of a mapping struct are loaded can be specified on a link definition with the :preload option, which accepts for now only a :depth option keyword with an integer for the preloading depth. The default value for :depth is 1. This means all of the data and object properties are loaded, including the nested Grax.Schema mapping with the descriptions of a linked resource, BUT NOT the linked Grax.Schema structs of these nested Grax.Schema structs. This essentially overwrites the preloading depth specification of the parent resource.

Instead of writing out the depth tuples there are some shortcuts for common depth values:

  • the boolean true is equivalent to {add_depth, 1}
  • the boolean false is equivalent to {add_depth, 0} and can be used to explicitly prohibit the preloading
  • non-additive depths can be specified with the integer for the depth level directly
  • additive depths can be specified in macro-contexts (on the preload opt of a link specification in the schema or on use Grax) with the an explicitly plus-signed integer like +1 for the depth level directly

Sometimes we want to define a link on a Grax.Schema for which no RDF property exists directly. For example, in our data there is no property linking a user to a post directly. Instead there is the schema:authorproperty which links a post to its authors, so exactly the inverse property of what we want. You can specify a link property on a Grax.Schema in those cases by declaring it as an inverse property with a minus sign before the IRI of the inverse property.

defmodule User do
  use Grax

  alias NS.{SchemaOrg, FOAF}

  schema do
    property name: SchemaOrg.name, type: :string, required: true
    property email: SchemaOrg.email, type: [:string], required: true
    property age: FOAF.age, type: :integer
    property password: nil
    
    link friends: FOAF.friend, type: [User]
    link posts: -SchemaOrg.author, type: [Post]
  end
end

defmodule Post do
  use Grax

  alias NS.SchemaOrg

  schema do
    property title: SchemaOrg.name(), type: :string
    property content: SchemaOrg.articleBody(), type: :string

    link author: SchemaOrg.author(), type: Example.User
  end
end
